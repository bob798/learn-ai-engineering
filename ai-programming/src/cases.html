OMC 赋能 PM 设计音频客服系统
产品经理利用多 Agent 编排，加速音频客服产品的设计全流程
L4 金丹境
🎯
SCENARIO
产品经理利用 oh-my-claudecode（OMC）的多 Agent 编排能力，面向电商售后场景设计音频客服系统，目标是替代 60% 的人工客服通话。
OMC 产品通道 Agent
Agent 能力 PM 怎么用
product-manager 需求拆解、PRD 生成、优先级排序 输入模糊想法，输出结构化需求文档
ux-researcher 用户画像、场景分析、竞品调研 分析音频客服的用户旅程和痛点
information-architect 信息架构、流程图、状态机设计 设计客服对话流转和状态机
product-analyst 数据分析、指标定义、ROI 测算 定义客服系统的核心指标和成本模型
辅助角色
researcher（深度调研） · architect（技术架构建议） · writer（文档输出） · critic（对抗式审核）
四阶段操作流程
1
需求探索 — Team 模式
三个 Agent 管线式协作，前一个的输出自动流入后一个。PM 不需要分别跟每个 Agent 对话。
Phase 1 示例 ▶
# PM 输入 "我要设计一个音频客服系统，面向电商售后场景， 目标是替代 60% 的人工客服通话" # OMC 编排 ux-researcher → 输出用户画像 + 场景地图 + 竞品分析 product-analyst → 输出市场规模 + 成本对比（人工 vs AI） product-manager → 综合两者，输出需求优先级矩阵
2
架构设计 — Team 模式
PM 不懂技术细节，但 architect Agent 会用 PM 能理解的语言解释技术方案的取舍。
Phase 2 示例 ▶
# PM 输入 "基于上面的需求，设计客服系统的对话流程和角色架构" # OMC 编排 information-architect → 输出对话状态机 + 角色分工图 architect → 输出技术架构建议（ASR → NLU → Agent → TTS） product-manager → 审核两者，标注产品风险和 MVP 边界
3
PRD 生成 — Autopilot 模式
需求明确时，不需要多 Agent 协作。product-manager 在 Autopilot 模式下自主生成完整 PRD。
Phase 3 示例 ▶
# PM 输入 "把前两轮的结论整理成 PRD，包含用户故事、验收标准、 技术约束和 MVP 范围" # OMC 编排 product-manager → 自主生成完整 PRD （Autopilot 模式：需求明确，不需要多 Agent 协作）
4
评审预演 — Ralph 模式
持续验证循环。每个 Agent 都带着「挑毛病」的视角审核，PM 在真正上评审会之前就修补了漏洞。
Phase 4 示例 ▶
# PM 输入 "帮我预演评审会，找出 PRD 里的漏洞" # OMC 编排 product-analyst → 挑战指标定义（"60% 替代率怎么衡量？"） ux-researcher → 挑战用户体验（"老年用户语音识别准确率？"） architect → 挑战技术可行性（"实时语音延迟能控制在多少？"） critic → 综合审核，输出风险清单
PM 典型命令示例
示例 1 — 竞品调研 ▶
"帮我调研音频客服赛道，重点关注： 1. 现有方案（讯飞/阿里云/AWS Connect）的优劣 2. 用户最痛的 3 个问题 3. AI 客服 vs 人工客服的成本结构" → OMC 自动调度 researcher + product-analyst + ux-researcher
示例 2 — 对话流程设计 ▶
"设计一个客服对话的状态机，覆盖： 问候 → 意图识别 → 问题处理 → 满意度确认 → 结束" → OMC 调度 information-architect 输出 Mermaid 流程图
示例 3 — 对抗式审核 ▶
"这个 PRD 有什么漏洞？假设你是技术总监，你会问什么？" → OMC 调度 critic + architect 进行对抗式审核
效率对比表
PM 工作环节 没有 OMC 有 OMC
竞品调研 自己搜索整理，2-3 天 researcher Agent 输出结构化报告，2-3 小时
用户画像 找 UX 同事排期，1-2 周 ux-researcher Agent 基于行业数据生成初版，PM 审核修改
对话流程设计 画 Visio/Figma 反复改，1 周 information-architect 输出状态机，PM 调整业务规则
PRD 撰写 写 3-5 天，格式和完整性参差 product-manager Agent 生成结构化 PRD，PM 填充判断和决策
评审预演 找同事模拟，时间难协调 critic Agent 随时可用，多视角对抗式审核
技术可行性判断 等研发评估，1-2 周 architect Agent 即时给出技术约束和风险
核心价值
调研加速
researcher / ux-researcher / product-analyst 把信息收集和结构化的脏活干了，PM 专注于判断和决策。
设计协作
information-architect / architect 提供专业视角，PM 不需要等排期就能获得多角色反馈。
质量防线
critic Agent 在评审前找漏洞，Ralph 模式持续验证，PM 带着更完善的方案上会。
一句话总结
OMC 让 PM 拥有了一支随时待命的虚拟产品团队，从调研到 PRD 到评审预演，全流程都有专业角色配合。

案例分析模板
拆解任何 AI 工具应用案例的四步框架
可复用
💡
HOW TO USE
拿到一个 AI 工具应用案例（无论是别人的还是自己的），用这四步拆解，提炼出可以迁移到你自己项目的洞察。
1
目标 — 它解决了什么问题？
明确案例的痛点和目标指标。

问自己：
• 没有 AI 工具之前，这件事怎么做的？耗时多久？
• 引入 AI 工具后，具体改善了什么环节？
• 用什么指标衡量成功？（时间、成本、质量、覆盖率……）
2
架构 — 系统是怎么设计的？
拆解角色分工和信息流向。

问自己：
• 涉及哪些角色（Agent / 人 / 工具）？每个角色的职责边界在哪？
• 信息是怎么流动的？谁产出，谁消费，谁决策？
• 用了哪种编排模式？（顺序 / 并行 / 管线 / 循环）
3
关键决策 — 做了哪些权衡？
找出案例中的取舍点和边界条件。

问自己：
• 为什么选这个方案，而不是更简单或更复杂的？
• 有没有刻意不做的事？（MVP 边界在哪？）
• 如果规模扩大 10 倍，这个设计会在哪里撑不住？
4
经验 — 什么可以复用到你的项目？
从案例中提炼可迁移的模式。

问自己：
• 这个案例中哪个角色分工方式可以直接借鉴？
• 哪个编排模式适合我当前项目的某个环节？
• 有什么坑是案例里踩过的，我可以提前绕开？
用音频客服案例套模板
目标
PM 独自设计复杂系统效率低，排期依赖重。目标：用 OMC 把竞品调研从 3 天缩到 3 小时，PRD 撰写从 5 天缩到 1 天。
架构
四个专业 Agent（PM / UX / IA / PA）+ 四个辅助角色，Team 模式管线协作 → Autopilot 模式生成文档 → Ralph 模式持续验证。
关键决策
PM 只做「判断和决策」，Agent 做「信息收集和结构化」。刻意不让 Agent 替代 PM 的业务判断。
经验
「对抗式审核」模式（Ralph + critic）可以直接复用到任何需要提前发现漏洞的场景：代码审查、方案评审、风险自查。
