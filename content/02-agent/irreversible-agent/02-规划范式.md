# Agent怎么"想"——走一步看一步，还是先做攻略？
## 《不可逆的Agent》系列 · 第2篇

> 本系列回答一个问题：当Agent的动作有真实世界后果时，该怎么设计。

---

你旅游时是哪一派？

一派人出发前做满Excel攻略，几点到哪、吃什么、备选方案是什么，全部想好。另一派人订张机票就走，落地再说，走到哪算哪。

两派谁也说服不了谁——因为这根本不是对错问题，而是一个取舍问题。而这个取舍，恰好就是Agent设计里最核心的那个决定。

## 打开那个黑盒

第1篇我们搭了一个最小Agent：LLM + 读写工具 + 循环 + 累积历史。其中有一行代码，我们当时当黑盒直接用了：

```python
action = llm.think(goal, tools, history)
```

"想"就发生在这一行：把当前状况和目标，变成下一步动作。这一篇就是把这个黑盒打开。

打开之后你会发现，里面藏着一个所有Agent框架都必须回答的问题：**想多少，再动手？**

## 翻车现场

先看不打开会发生什么。给第1篇那个最小Agent派个更大的任务：

"优化全屋照明——有人的房间保持舒适亮度，没人的房间关灯。"

八个房间。跑起来之后，日志大概是这样的：

```
第1轮：查客厅状态 → 亮度80%，有人
第2轮：客厅调到50%
第3轮：查卧室 → 没人，灯开着
第4轮：关卧室灯
第5轮：查客厅状态        ← 等等，客厅不是查过了？
第6轮：查书房 → 没人
第7轮：书房灯调到30%      ← 没人为什么不是关掉？
第8轮：查厨房……
```

三个毛病肉眼可见。**它兜圈子**——第5轮又查了一遍客厅，因为历史越来越长，它"看漏"了自己做过什么。**它目标漂移**——第7轮忘了"没人就关灯"的规则，做了个自作主张的调整。**它很贵**——八个房间，十几轮循环，每一轮都是一次完整的LLM调用，token账单随着历史累积越滚越大。

这个走一步看一步的模式，有个正式的名字。

## ReAct：边想边做

ReAct（Yao et al., 2022）是让工具型Agent真正可用的奠基模式，名字就是它的全部：Reason + Act。循环是 思考→动作→观察→再思考，每一轮都基于最新观察决定下一步，直到它认为信息够了、任务完成了。

第1篇的最小Agent天然就是ReAct——我们没有刻意设计，它就是这个形态。这不是巧合：ReAct是Agent的"默认物理形态"，实现最简单，而且**随机应变**——环境返回什么意外，它下一轮就能接住。

但翻车现场暴露了它的代价：**短视**。它永远只想下一步，没有全局图景，所以会绕路、会重复、会在长任务里慢慢忘掉最初为什么出发。就像没有攻略的旅行者，自由，但容易在同一条街上走三遍。

那反过来，先想好再动手呢？

## Plan-and-Execute：先做攻略

另一种活法：接到任务，先不碰任何工具，而是生成一份完整计划——

```
计划：优化全屋照明
1. 查询全部8个房间的状态（占用+亮度）
2. 有人的房间：调整到50%舒适亮度
3. 没人的房间：关灯
4. 复查全部房间，确认符合目标
```

然后按计划逐条执行。执行器不需要每步都"深度思考"，照单干活就行——规划用强推理，执行用便宜的调用，成本立刻降下来。

这不是纸上谈兵，你可能每天都在用它。Claude Code——目前最主流的编码Agent——官方推荐的工作流就是"探索→规划→实施→提交"四阶段，并且给了一条经验法则：如果一个改动你能用一句话说清，跳过规划；说不清，先规划。执行时它会自动创建todo list，每个条目有pending、in_progress、completed三种状态，实时追踪走到了哪一步。

规划的收益是可测量的。一个安全攻防（CTF）实验里，纯ReAct的Agent解题成功率83%；仅仅加上一个规划步骤、并给规划用更强的模型，成功率提到了95%。规划不是玄学。

而且注意一个安静但重要的细节：**那份计划，在执行前是可以被审查的。**你在Agent碰任何东西之前，就看到了它打算做什么——"第3步要关灯"白纸黑字摆在那。这一点现在看只是个透明度福利，先记住它。

## 攻略派的死穴

那为什么不全用plan-and-execute？

因为**计划赶不上变化**。执行到第3步，有人走进了书房——计划里"书房没人，关灯"这一条瞬间作废。世界变了，攻略还是旧的。

所以实用的plan-and-execute必须带一个逃生舱：**重规划**。执行中一旦发现现实和计划的前提不符，回到规划步，基于新状况重新出攻略。计划不是圣旨，是随时可以推翻重写的草稿。

但重规划本身有成本，环境变得越频繁，重规划越多，它就越退化成一个昂贵版的ReAct。于是问题回到了原点——

## 取舍：那条分界线在哪

把两派放在一起，判断维度其实就三个：

**任务可预测性。**步骤越能提前确定，越该规划。"优化全屋照明"步骤清晰，适合攻略派；"帮我查清楚这个bug出在哪"充满未知，适合随缘派。

**环境动态性。**世界变得越快，攻略作废越快。家里人员频繁走动，规划的保质期就短；深夜无人的办公楼，一份计划能执行到底。

**出错代价。**这是最容易被忽略、也最重要的一条。错一步无所谓的任务，走一步看一步就好；错一步很贵的任务，你会希望在动手之前就看到完整计划——因为**可审查性只有攻略派才有**。

Claude Code那条"一句话说不清就先规划"的法则，本质是第一条维度的极简版。实践中的成熟形态往往是混合：粗粒度先规划立骨架，每个步骤内部留一点ReAct式的随机应变。

## 伏笔：当"可审查"从福利变成生死线

现在回收前面埋的那个细节。

在软件世界，"计划可被审查"是锦上添花——反正做错了能回滚，审不审查，代价差不太多。所以软件Agent选范式，主要看前两条维度：可预测性和动态性。

但把场景换掉：Agent操作的不是代码，是真实的灯光、阀门、电机。

第三条维度——出错代价——突然压倒了一切。一个走一步看一步的Agent操作物理设备，意味着**没有人在任何时刻知道它接下来要干什么**，等于边开车边发明路。而一份执行前的完整计划，是人类或安全系统唯一的介入窗口：在动作发生之前，过目、拦截、修改。

这就是为什么，当Agent的手伸进物理世界，范式的天平几乎注定倒向plan-and-execute——计划不再是效率优化，而是给Agent装护栏的入口：一份执行前的完整计划，可以逐条对着护栏检查，危险的那一步在它发生之前就被拦下。至于这套护栏（专业上叫"安全包络"，Safety Envelope）具体怎么设计，是第8篇的全部内容。

## 留一个问题

回到开头的旅游。

有意思的是，大多数人旅游是攻略派，人生却是随缘派——我们似乎凭直觉就知道，什么事值得提前想清楚，什么事走着看就行。

那条直觉的分界线，你能说清楚它是什么吗？下次给Agent选"想"的方式时，你画的那条线，和你给自己画的，是同一条吗？
