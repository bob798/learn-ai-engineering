Agent 怎么"碰"世界——从 function calling 到 MCP

不可逆的 Agent · 第 4 篇

本系列回答一个问题：当 Agent 的动作有真实世界后果时，该怎么设计。

前三篇，我们给 Agent 装了脑子（推理）、给了它记忆、教它规划。但有一件最基本的事，我们一直当黑盒用着从没打开：

它到底是怎么碰到那盏灯的？

第 1 篇我说"给 Agent 一个 set_light 工具，它就能关灯了"。可一个语言模型——一个只会输出文字的东西——是怎么让一盏真实的灯熄灭的？这一篇就是把这个黑盒彻底打开。打开之后你会看到一个反直觉的真相，而这个真相，恰好是整个物理 Agent 安全体系的地基。

· · ·

第一层真相：LLM 其实碰不到任何东西

先破除一个几乎所有人都有的误解。

我们总说"Agent 自己在操作""模型调用了工具"，这话给人一种画面：模型像一只手，伸出去把灯关了。

这个画面是错的。 LLM 在物理上就没有"手"。它是一个文本生成模型，输入文字、输出文字，仅此而已。它做不了任何别的事。

那它怎么关灯？答案是：它不关灯，它只是写了一张纸条。

当你让它"把客厅灯关掉"，它真正做的，是生成一段结构化的文字，大意是：

{ "调用": "set_light", "参数": { "房间": "客厅", "状态": "关" } }

这段文字本身关不了任何灯。真正去关灯的，是外面的一段代码——它读到这张纸条，解析出"哦，模型想调 set_light"，然后由它去执行真正的操作。

LLM 从不调用，它只是生成了一个调用请求。
提议的是模型，动手的永远是外面的代码。

记住这个"提议与执行的分离"。它现在看只是个技术细节，到文章结尾它会变成整篇的命门。

· · ·

为什么需要这层"纸条"——function calling

你可能会问：为什么要这么绕？让模型直接说人话"帮我把灯关了"，不行吗？

不行，因为外部代码没法可靠地理解人话。"把灯关了""关一下灯""灯不用开着"——同一个意图有一万种说法，代码没法稳定地从自由文本里解析出"到底调哪个函数、填什么参数"。

于是需要一种约定：让模型把"我想干什么"表达成一种机器能稳定解析的结构化格式。这个能力，就叫 function calling（函数调用）。它是模型被专门训练出来的一项本领——不说人话，而是吐出规整的、带名字和参数的调用请求。

function calling 是 LLM 从"只会聊天"到"能表达可执行意图"的翻译层。有了它，模型的想法才能被外部世界接住。

· · ·

厘清三个总被搞混的词：function calling、tool、MCP

这三个词经常混用，但它们是三个不同层次的东西。用一个餐厅的比方一次讲清：

  • Tool（工具）是"菜"。一个工具 = 一个函数（比如 set_light）+ 它的真实实现 + 一段给模型看的描述。
  • Function calling 是"点菜的能力"。是模型从菜单里"点选"某道菜的机制。
  • MCP 是"标准化的菜单格式"。它规定菜单该怎么写，好让任何餐厅的菜单，任何顾客都看得懂。

所以它们不是竞品，是一条链上的三环：你定义 tool，模型用 function calling 去点选，MCP 让这套菜单标准化。

· · ·

模型怎么知道该点哪道菜？

答案全在描述里。你每次请求时，会把"可用工具清单 + 每个工具的描述"一起喂给模型。模型读这份清单来做两个判断：

  • 要不要用工具："法国首都是哪"——直接答；"客厅现在几度"——清单里有个 get_temp 说能查，那就调它。
  • 怎么用：靠工具描述里的参数说明，知道要填一个 room、值是房间名。

所以工具描述写得好不好，直接决定模型用得对不对。给工具写描述，本质是在给一个新员工写操作手册——这是 Agent 工程里一个被严重低估的手艺。

· · ·

工具一多，就掉进 N×M 的地狱

真实世界里，你想让 Agent 接入 Gmail、GitHub、日历、灯光系统、数据库……几十上百个。难点不在数量，在每个工具说的话都不一样。

这就是 N×M 问题：N 个工具 × M 个模型 = N×M 套适配代码。在 MCP 出现之前，如果你有 10 个 AI 应用和 100 个工具，可能需要写 1000 个不同的集成。

常见误解："那大家都用 HTTP 不就统一了？" 没用。HTTP 只统一了"怎么把字节传过去"，没统一"传的内容长什么样"。就像大家都用声波说话，不代表中国人和法国人能互相听懂。N×M 是应用层的语义分裂，传输层的统一解决不了。

· · ·

MCP：给所有工具装一个通用插头

解药是：别让每个工具各说各话，让它们说同一种语言。

这就是 MCP（Model Context Protocol，模型上下文协议）。2024 年 11 月由 Anthropic 提出，官方比喻是"AI 应用的 USB-C"：以前每种设备一种接口，现在一个通用口，插上就能用。

效果是把 N×M 塌缩成 N+M：每个工具实现一次 MCP server，每个模型实现一次 MCP client，然后自由组合。集成成本从"相乘"变成"相加"。

MCP 不是 HTTP 的替代品，是驾在 HTTP 之上、专门给模型和工具沟通用的一层语言。HTTP 是邮路，MCP 是信里的标准公文格式。

而且这不再是"一种新兴方案"。2025 年 12 月，Anthropic 把 MCP 捐给了 Linux 基金会，OpenAI、Google、微软共同参与——它已经从"某一家的协议"变成了跨厂商的行业标准。

· · ·

但是——我们只讲了"碰出去"，没讲"碰的反馈"

这条链是单向的——它只讲了指令怎么发出去。可一次真正的"碰"，是双向的：动作发出去之后，结果怎么回来？

你发出"关灯"，然后呢？灯真的关了吗？返回的"成功"可信吗？在物理世界，"模型以为灯关了"和"灯真的关了"之间，隔着一整个真实世界的不确定性。

碰世界不只是碰出去，更是接住碰回来的反馈。这个"结果回传与确认"，是后面某一篇的全部内容。

· · ·

命门：当"接入"变得太容易

现在回到开头那个让你记住的东西：提议与执行的分离。

MCP 让接入变得前所未有地简单——2026 年已有上万个 MCP server、SDK 月下载量以千万计。任何能力，包装成一个 MCP server，就能插进任何 Agent。

但请把这句话念慢一点：任何能力，都能轻松插进去。

查天气的能力，无所谓。可如果插进去的是"删除数据库""转账""开门锁""打开阀门"呢？MCP 不区分这些——在协议眼里，关灯和转账都只是一次工具调用。接入的门槛降到了地板上，但风险的门槛没有跟着降。

这不是危言耸听。随着 MCP 铺开，安全事故接踵而至：跨租户数据泄露、工具投毒攻击、两个月内三十多个漏洞。行业被迫补课——把认证授权标准化、给敏感操作加上"人在环中"的审批。

正因为 LLM 只能"提议"、不能"执行"，
中间那层代码才成了唯一能把关的地方——
在动作真的发生之前，检查它、拦截它、或者交给人确认。

把同样的逻辑挪到物理世界：Agent 接入的不再是数据库，是电机、阀门、门锁。一次错误的工具调用，另一端就是一扇真实打开的门。而物理世界，没有回滚。

所以物理 Agent 的工具层，比软件 Agent 多一道无法省略的关：问题不再是"能不能接入"，而是"接入之后，谁来为每一次调用把关"。那道关卡怎么设计——是第 8 篇的全部内容。

· · ·

留一个问题

你给 Agent 接的每一个工具，本质上都是在授予它一项能力。

能力的门槛，和风险的门槛，从来不是同一道。而我们造的工具，正在让前一道飞速降低。那么，守住后一道的，该是谁？
