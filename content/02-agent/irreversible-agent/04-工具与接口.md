# Agent怎么"碰"世界——从function calling到MCP
## 《不可逆的Agent》系列 · 第4篇

> 本系列回答一个问题：当Agent的动作有真实世界后果时，该怎么设计。

---

前三篇，我们给Agent装了脑子（推理）、给了它记忆、教它规划。但有一件最基本的事，我们一直当黑盒用着从没打开：

它到底是**怎么碰到那盏灯的**？

第1篇我说"给Agent一个`set_light`工具，它就能关灯了"。可一个语言模型——一个只会输出文字的东西——是怎么让一盏真实的灯熄灭的？这一篇就是把这个黑盒彻底打开。打开之后你会看到一个反直觉的真相，而这个真相，恰好是整个物理Agent安全体系的地基。

## 第一层真相：LLM其实碰不到任何东西

先破除一个几乎所有人都有的误解。

我们总说"Agent自己在操作""模型调用了工具"，这话给人一种画面：模型像一只手，伸出去把灯关了。

**这个画面是错的。** LLM在物理上就没有"手"。它是一个文本生成模型，输入文字、输出文字，仅此而已。它做不了任何别的事。

那它怎么关灯？答案是：**它不关灯，它只是写了一张纸条。**

当你让它"把客厅灯关掉"，它真正做的，是生成一段结构化的文字，大意是：

```json
{ "调用": "set_light", "参数": { "房间": "客厅", "状态": "关" } }
```

这段文字本身关不了任何灯。真正去关灯的，是**外面的一段代码**——它读到这张纸条，解析出"哦，模型想调set_light"，然后由它去执行真正的操作。

所以"LLM调用工具"这句话有严重的误导性。**LLM从不调用，它只是生成了一个调用请求。** 提议的是模型，动手的永远是外面的代码。

记住这个"提议与执行的分离"。它现在看只是个技术细节，到文章结尾它会变成整篇的命门。

## 为什么需要这层"纸条"——function calling

你可能会问：为什么要这么绕？让模型直接说人话"帮我把灯关了"，不行吗？

不行，因为**外部代码没法可靠地理解人话**。"把灯关了""关一下灯""灯不用开着"——同一个意图有一万种说法，代码没法稳定地从自由文本里解析出"到底调哪个函数、填什么参数"。

于是需要一种约定：让模型把"我想干什么"表达成一种**机器能稳定解析的结构化格式**。这个能力，就叫 **function calling（函数调用）**。它是模型被专门训练出来的一项本领——不说人话，而是吐出规整的、带名字和参数的调用请求。

**function calling是LLM从"只会聊天"到"能表达可执行意图"的翻译层。** 有了它，模型的想法才能被外部世界接住。

## 厘清三个总被搞混的词：function calling、tool、MCP

这三个词经常混用，但它们是三个不同层次的东西。用一个餐厅的比方一次讲清：

- **Tool（工具）是"菜"。** 一个工具 = 一个函数（比如`set_light`）+ 它的真实实现 + 一段给模型看的描述。是你（开发者）准备好、摆上菜单的东西。
- **Function calling是"点菜的能力"。** 是模型从菜单里"点选"某道菜的机制。菜是你备的，点菜的本事是模型自带的。
- **MCP是"标准化的菜单格式"。** 它规定菜单该怎么写，好让任何餐厅的菜单，任何顾客都看得懂。

所以它们不是竞品，是一条链上的三环：你定义**tool**，模型用**function calling**去点选，**MCP**让这套菜单标准化。下面分别看后两环。

## 模型怎么知道该点哪道菜？

一个关键问题：模型怎么知道"此刻该调工具"而不是自己瞎答？又怎么知道该填什么参数？

答案全在**描述**里。你每次请求时，会把"可用工具清单 + 每个工具的描述"一起喂给模型。模型读这份清单来做两个判断：

- **要不要用工具**："法国首都是哪"——我自己知道，直接答；"客厅现在几度"——我的训练数据里没有实时温度，但清单里有个`get_temp`说能查，那就调它。这个判断是模型在训练中学会的能力。
- **怎么用**：靠工具描述里的参数说明，知道要填一个`room`、值是房间名。

所以**工具描述写得好不好，直接决定模型用得对不对**。描述含糊，模型就会乱填参数、或者该用的时候不用。给工具写描述，本质是在给一个新员工写操作手册——这是Agent工程里一个被严重低估的手艺。

## 工具一多，就掉进N×M的地狱

现在问题来了。上面说的是一两个工具。真实世界里，你想让Agent接入Gmail、GitHub、日历、你的灯光系统、你的数据库……几十上百个。

难点不在数量，在**每个工具说的话都不一样**。Gmail的接口一套规矩，GitHub的接口另一套规矩，参数命名、鉴权方式、错误格式全不同。你的Agent想接Gmail，得写一套适配Gmail的代码；想接GitHub，再写一套。

更糟的是，模型不止一个。Claude、GPT、你自研的Agent……每一个模型应用，都要为每一个工具重写一遍适配。

这就是 **N×M 问题**：N个工具 × M个模型 = N×M套适配代码。在MCP出现之前，如果你有10个AI应用和100个工具，可能需要写1000个不同的集成。这是整个行业当时真实的泥潭。

（顺带说清一个常见误解：**"那大家都用HTTP不就统一了？"** 没用。HTTP只统一了"怎么把字节传过去"，没统一"传的内容长什么样"。每个API照样各说各话——就像大家都用声波说话，不代表中国人和法国人能互相听懂。N×M是应用层的语义分裂，传输层的统一解决不了。）

## MCP：给所有工具装一个通用插头

解药是：**别让每个工具各说各话，让它们说同一种语言。**

这就是 MCP（Model Context Protocol，模型上下文协议）。2024年11月由Anthropic提出，它做的事很简单——规定一套统一的约定：工具怎么自我描述、怎么声明参数、怎么被发现、怎么返回结果。任何工具只要按MCP实现一个"server"，任何模型只要会说MCP当"client"，二者就能对接，不用再写专门的适配。

官方给它的比喻是"**AI应用的USB-C**"：以前每种设备一种接口线缆，现在一个通用口，插上就能用。

效果是把 **N×M 塌缩成 N+M**：每个工具实现一次MCP server，每个模型实现一次MCP client，然后所有工具和所有模型自由组合。集成成本从"相乘"变成"相加"。

这里可以印证你前面那个直觉——**MCP不是HTTP的替代品，是驾在HTTP之上、专门给模型和工具沟通用的一层语言。** 事实上MCP即将发布的新版规范，核心改进之一正是让它在协议层变得更"无状态"，从而能更好地跑在普通HTTP基础设施上。它的演进方向是"更好地利用HTTP"，而非取代它。HTTP是邮路，MCP是信里的标准公文格式。

而且这不再是"一种新兴方案"。2025年12月，Anthropic把MCP捐给了Linux基金会，OpenAI、Google、微软共同参与——它已经从"某一家的协议"变成了跨厂商的行业标准。写下这篇时，它就是既成事实。

## 但是——我们只讲了"碰出去"，没讲"碰的反馈"

到这里，"Agent怎么碰世界"好像讲完了：模型提议 → function calling表达 → tool承接 → MCP标准化 → 执行。

但请注意，这条链是**单向**的——它只讲了指令怎么**发出去**。可一次真正的"碰"，是双向的：动作发出去之后，**结果怎么回来？**

你发出"关灯"，然后呢？灯真的关了吗？返回的"成功"可信吗？如果设备根本没响应呢？

在软件世界，这一环常被无视——函数返回了就算成了。但你已经能预感到，在物理世界这会成为致命问题："模型以为灯关了"和"灯真的关了"之间，隔着一整个真实世界的不确定性。这个"结果回传与确认"，是后面某一篇（状态验证）的全部内容。这里先把这根线头留出来——**碰世界不只是碰出去，更是接住碰回来的反馈。**

## 命门：当"接入"变得太容易

现在回到开头那个让你记住的东西：**提议与执行的分离。**

MCP的伟大之处，是让接入一个工具变得前所未有地简单——简单到2026年已经有上万个MCP server、SDK月下载量以千万计。任何能力，包装成一个MCP server，就能轻松插进任何Agent。

但请把这句话念慢一点：**任何能力，都能轻松插进去。**

查天气的能力，插进去无所谓。可如果插进去的是"删除数据库""转账""开门锁""打开阀门"呢？MCP不区分这些——在协议眼里，关灯和转账都只是一次工具调用。**接入的门槛降到了地板上，但风险的门槛没有跟着降。**

这不是危言耸听，是2026年正在真实上演的事。随着MCP铺开，安全事故接踵而至：有厂商出现跨租户的数据泄露，有平台被"工具投毒"攻击，短短两个月安全研究者提交了三十多个漏洞。行业被迫补课——把认证授权标准化、给敏感操作加上"人在环中"的审批。有公司（比如Pinterest）的生产实践已经明确：领域专用的工具server，加上对敏感操作的人工审批环节，缺一不可。

看出来了吗？行业正在用真实的事故，重新发现那个被忽略的分离层的价值。**正因为LLM只能"提议"、不能"执行"，中间那层代码才成了唯一能把关的地方**——在动作真的发生之前，检查它、拦截它、或者交给人确认。接入越容易，这道关卡就越不能省。

而这，还只是软件世界的故事。软件里的"工具投毒"，最坏是泄露数据、丢钱——都还能补救。

把同样的逻辑挪到物理世界：Agent接入的不再是数据库，是电机、阀门、门锁、暖通主机。一次被投毒的工具调用、一个模型的错误提议，另一端就是一扇真实打开的门、一个真实开启的阀门。而物理世界，没有回滚。

所以物理Agent的工具层，比软件Agent多一道无法省略的关：问题不再是"能不能接入"（MCP已经让它太容易了），而是"**接入之后，谁来为每一次调用把关**"。那道关卡具体装什么、怎么给动作分级、护栏怎么设计——是第8篇的全部内容。

## 留一个问题

你给Agent接的每一个工具，本质上都是在授予它一项**能力**。

我们习惯了用"方不方便"来衡量要不要接一个工具——接入越简单越好。但如果这个工具能删库、能转账、能开锁，你还会像接一个查天气的工具那样，随手就插上吗？

**能力的门槛，和风险的门槛，从来不是同一道。** 而我们造的工具，正在让前一道飞速降低。那么，守住后一道的，该是谁？

### 参考与依据

**MCP的定位、起源与标准化：**

- Anthropic于2024年11月25日推出MCP，官方将其定位为"AI应用的USB-C"式通用连接协议。2025年12月，Anthropic将MCP捐赠给Linux基金会旗下的Agentic AI Foundation，OpenAI、Google、微软等共同参与治理，MCP由此成为跨厂商的中立行业标准。
- N×M → N+M：MCP官方与多方分析均以此描述其核心价值（"10个应用×100个工具需要1000个集成"降为"每资源一个server、任意客户端可用"）。
- MCP新规范（2026-07-28版，发稿时为候选状态）将协议核心改为无状态，以便运行在普通HTTP基础设施上——印证"MCP架于HTTP之上"的关系。来源：MCP官方博客。

**MCP安全事故与"人在环中"审批的兴起（本篇物理伏笔的现实依据）：**

- 2026年初，随MCP普及出现多起生产安全事件（跨租户数据泄露、路径遍历、工具投毒等），两个月内安全研究者提交30余个CVE。
- 2026年7月，MCP团队将"企业级托管授权"（Enterprise-Managed Authorisation）扩展升为稳定版，由Anthropic、Microsoft、Okta等采用，用集中式身份控制替代逐server授权。
- Pinterest等公司公开的生产MCP架构，对敏感操作明确采用human-in-the-loop（人在环中）审批。这些共同构成"能力易得之后，把关成为刚需"的行业例证。

**概念性说明：**

- function calling、tool、MCP的层次区分，以及"LLM只提议、不执行"的机制，属于对当前主流Agent架构的通识性归纳。
- 文中灯光Agent、JSON纸条等为示意性构造，用于说明机制。
