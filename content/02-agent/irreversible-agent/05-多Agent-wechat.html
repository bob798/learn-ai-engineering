不可逆的 Agent · 第 5 篇

本系列回答一个问题：当 Agent 的动作有真实世界后果时，该怎么设计。

前四篇，我们让一个 Agent 越来越强。顺着这个势头，下一步好像理所当然：多来几个，组队，分工协作。

这一篇泼盆冷水。多 Agent 是整个 AI 工程里最被高估的答案。

· · ·

数据先行

MAST 研究（7 个主流框架、1600+ 条真实轨迹，NeurIPS 2025）给出的数字：多 Agent 生产失败率 41%–87%。 绝大多数失败不是模型不聪明，是协作本身出了问题——角色误解、重复劳动、错误沿交接链污染全局。研究者的结论冷峻：光靠模型能力提升，不足以解决这一类失败。

更扎心的是：简单的单 Agent 方案，常以低得多的成本，追平甚至超过多 Agent 设计。 所以：多 Agent 不是默认选项，单 Agent 才是。

· · ·

为什么拆开反而更糟

协作有成本，且成本长得很快。n 个 Agent 自由通信，交互数是 n(n-1)/2——平方级。1 变 5，复杂度不是 5 倍，是 25 倍。

这根本不是 AI 问题——这是分布式系统问题，微服务架构的人十五年前就全趟过一遍了。一个 Agent 是功能，五十个 Agent 是"没人在认真讨论的分布式系统问题"。

· · ·

真要拆：三条原则

一、沿领域拆，不沿功能横切。 高内聚低耦合——灯光、暖通、安防各归各的 Agent，领域内自成一体，领域间少来往。按"感知/决策/执行"横切，干任何事都要三个 Agent 来回通信，耦合度爆表。

二、默认用编排（有总指挥），慎用协同。 只有 Agent 完全对等、任务并行只读、错误不污染下游时，才用协同。否则一律编排。协同在图上优雅，在故障中痛苦——出事时没有人知道全局状态。总指挥的价值不在于做认知工作，在于它始终知道系统做过什么。

三、冲突仲裁必须是确定性规则，不能交给 LLM。 "安全 > 舒适 > 节能"写死在系统里。让 LLM 去商量，只是加了一层会幻觉的不确定性。

另外记住一个词——补偿（Saga 模式）。它不是"撤销"，是执行一个方向相反的新动作去抵消后果。这个区别在软件里无所谓，到物理世界会变成生死之别。

· · ·

伏笔：物理世界拆分是为了"关住爆炸"

物理 Agent 有一个软件世界没有的拆分理由：故障隔离。如果灯光 Agent 疯了，你要确保它绝对碰不到暖通主机和门禁——给每个 Agent 画一个物理的牢笼，把爆炸半径锁死在它自己的域内。

MAST 研究还发现：多 Agent 的失败常常不留痕迹，从微小失误慢慢累积，没有清晰源头。在物理世界，一个无法追溯的错误，意味着你可能在它酿成不可逆后果之前，根本没机会发现。

所以物理 Agent 的分域，是一种悲观主义设计：它假设错误一定会发生且可能发现不了，于是提前用物理边界锁死每个 Agent 能造成的最大伤害。
