# 别急着上多Agent——一个被高估的答案
## 《不可逆的Agent》系列 · 第5篇

> 本系列回答一个问题：当Agent的动作有真实世界后果时，该怎么设计。

---

前四篇，我们让一个Agent越来越强。顺着这个势头，下一步好像理所当然：多来几个，组队，分工协作。

这一篇泼盆冷水。**多Agent是整个AI工程里最被高估的答案。**

## 数据先行

MAST研究（7个主流框架、1600+条真实轨迹，NeurIPS 2025）给出的数字：**多Agent生产失败率41%–87%。** 绝大多数失败不是模型不聪明，是**协作本身出了问题**——角色误解、重复劳动、错误沿交接链污染全局。研究者的结论冷峻：光靠模型能力提升，不足以解决这一类失败。

更扎心的是：**简单的单Agent方案，常以低得多的成本，追平甚至超过多Agent设计。** 所以：**多Agent不是默认选项，单Agent才是。**

## 为什么拆开反而更糟

协作有成本，且成本长得很快。n个Agent自由通信，交互数是n(n-1)/2——**平方级**。1变5，复杂度不是5倍，是25倍。

这根本不是AI问题——**这是分布式系统问题**，微服务架构的人十五年前就全趟过一遍了。一个Agent是功能，五十个Agent是"没人在认真讨论的分布式系统问题"。

## 真要拆：三条原则

**一、沿领域拆，不沿功能横切。** 高内聚低耦合——灯光、暖通、安防各归各的Agent，领域内自成一体，领域间少来往。按"感知/决策/执行"横切，干任何事都要三个Agent来回通信，耦合度爆表。

**二、默认用编排（有总指挥），慎用协同。** 只有Agent完全对等、任务并行只读、错误不污染下游时，才用协同。否则一律编排。协同在图上优雅，在故障中痛苦——出事时没有人知道全局状态。总指挥的价值不在于做认知工作，在于**它始终知道系统做过什么**。

**三、冲突仲裁必须是确定性规则，不能交给LLM。** "安全 > 舒适 > 节能"写死在系统里。让LLM去商量，只是加了一层会幻觉的不确定性。

另外记住一个词——**补偿**（Saga模式）。它不是"撤销"，是执行一个方向相反的新动作去抵消后果。这个区别在软件里无所谓，到物理世界会变成生死之别。

## 伏笔：物理世界拆分是为了"关住爆炸"

物理Agent有一个软件世界没有的拆分理由：**故障隔离**。如果灯光Agent疯了，你要确保它绝对碰不到暖通主机和门禁——**给每个Agent画一个物理的牢笼**，把爆炸半径锁死在它自己的域内。

MAST研究还发现：多Agent的失败常常**不留痕迹**，从微小失误慢慢累积，没有清晰源头。在物理世界，一个无法追溯的错误，意味着你可能在它酿成不可逆后果之前，根本没机会发现。

所以物理Agent的分域，是一种悲观主义设计：它假设错误**一定会**发生且**可能发现不了**，于是提前用物理边界锁死每个Agent能造成的最大伤害。

## 留一个问题

**我是真的需要一支军队，还是只是没把手里这一个人用好？** 如果真需要——你能不能把一个士兵的错误，关在他一个人的房间里？
