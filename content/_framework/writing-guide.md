# Writing Guide · 文章编写规范

> 这份规范是从一篇事故复盘文的反复打磨里攒出来的（实例见 `content/02-agent/harness/agent-sandbox-lifecycle-reliability.md`）。下面每条都是当时踩出来的，不是凭空立的。写新文章或改旧文时照着走。

---

## 0. 一句话

开头就把核心挖出来，结构要有脊梁别平铺，引用分清事实和观点，文字说人话别端着，最后再砍掉三成废话。

---

## 1. 流程：先给结构，再动手

- 改结构、大改之前，先把"改成什么样、为什么"列出来给作者过目，批准了再落地。别闷头改完一大篇才发现方向错了。
- 沟通直接给判断，不堆一堆选项让人选。

## 2. 结构：让读者快速看到核心

- **开头就钻到核心**，别让人读到一半才摸到重点。两个好用的开头：第一性原理（把问题剥到最底层那一两条道理），或者五个为什么（顺着现象往下问五层，问到根上）。
- **别同质化**。最常见的毛病是整篇都是平铺的要点列表，每节长得一样。要有一条主线串起来（比如：钻到核心 → 问题本质 → 最该改的地方 → 怎么修 → 沉淀），每节换一种形态：状态机图、四象限表、对比矩阵、横向对照表、检查清单，别全是 bullet。
- **最尖锐的那个洞察，放显眼处**，别埋在中间某一节里。

## 3. 文字：说人话，去 AI 味

写得像工程师当面聊，不端着。下面这些是"AI 味"的典型标志，能去就去：

- 加粗满天飞 → 一段最多一个重点，大部分别加粗。
- 破折号「——」反复制造反转停顿 → 改成正常句子。
- 金句、标语、贴标签（「第一性原理一击」「免费的可靠性」「心法：」「一句话：」这类）→ 删掉，把道理直接写成一句普通总结。
- 排比口诀（同一组词每段都念一遍）→ 念一次就够。
- 自我表扬式小标题（「这一幕要证明什么」「钻到底了」）→ 弱化或去掉。

## 4. 一致性：文案和结构都要统一

- **标题层级是一棵干净的树**：一个 H1（标题）→ 大节 H2 → 小节 H3。别出现多个 H1，别让某个小节游离在体系外。
- **同类的东西用同一种写法**：每节开头的引导、每节末尾的总结句，格式对齐。
- **清掉骨架残留**：「示例锚点：」「配图建议：」「钩子：」这种写提纲时的元语言，成稿里要么删、要么改写进正文，不能一半成品一半提纲。

## 5. 篇幅：砍废话

- 写完再过一遍，砍掉连接废话和重复，正文压到约七成。
- 表格、代码块、状态机图这类结构不算进压缩目标，别为了凑字数把它们也删了。

## 6. 引用与观点标识

技术写作最常见的问题是分不清「哪些是公认事实，哪些是作者自己的判断」。读者不知道一个结论是论文写的还是作者编的，信任度就上不去。

### 三类标识

在正文中用以下标识区分内容性质，让读者一眼看出每段话的可信度来源：

- 📚 **事实（Fact）**：引用论文、教材、历史资料，说明已有共识。
- 🔍 **分析（Analysis）**：基于多个来源的归纳和解释，不是某一篇论文的原文。
- 💡 **作者观点（Author's View）**：自己的第一性原理思考和判断。

不需要每段都标，只在性质切换时标一次。连续几段都是事实，第一段标一次就够。

### 引用怎么写

- 观点类文章采用「观点 → 依据 → 我的思考」三层结构：先亮判断，再给证据，最后说自己怎么看。不要把所有引用堆到文末，依据要跟着观点走。
- 引用论文或教材时，正文里用方括号编号（如 [1]），文末统一列参考资料。
- 直接引用原文要用引用块（`>`），注明作者、出处和年份。引用必须核实原文，不能凭印象意译后当原文用。
- 文末参考资料分类列出（如：经典理论 / LLM 基础 / Agent 关键工作），格式：作者 (年份). *标题*. 会议或出版社。arXiv 预印本要标注 arXiv 编号，不要混充会议论文。
- 开源项目、产品功能等非学术来源，正文里说清楚就行，不用编号进参考资料。

### 实例

见 `content/02-agent/concepts/why-agent-is-inevitable.md`——全文用三类标识 + 行内引用 + 文末参考资料的完整示范。

## 7. 取材自真实项目：脱敏

- 抹掉真实项目名、包名、路径、租户名，只留可公开的技术内核。
- 内部证据（私有 issue 编号、截图、抛栈）标清楚"私有"，正文只保留拆解结论。

## 用法

新写文章：按 §2 先定结构、§3 定调子、§6 定引用层次，写完用 §5 压一遍。

改旧文章：先用 §1 把改法讲清楚，再依次过 §2（结构）→ §4（一致性）→ §6（引用与标识）→ §3（去 AI 味）→ §5（压篇幅）。
